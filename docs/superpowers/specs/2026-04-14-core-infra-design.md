# Core Infrastructure Design

**Date:** 2026-04-14
**Scope:** Multi-cloud personal infrastructure repo — GCP baseline first, extensible to AWS, Hetzner, GitHub

---

## Overview

A Terraform monorepo managing personal hobby infrastructure across multiple cloud providers. Each provider is independently deployable with isolated remote state. GCP is the first provider to be fully built out.

## Section 1 — Bootstrap

**Purpose:** Creates the GCS bucket used as a remote state backend by all other Terraform roots. Run once manually.

**State:** Local (intentional — manages only one resource, run once).

**Provisions:**
- GCS bucket with versioning enabled and uniform bucket-level access
- Lifecycle rule to delete old state object versions after 90 days (cost control)

**Auth:** Personal ADC (`gcloud auth application-default login`)

**Variables (in gitignored `terraform.tfvars`):**
- `project_id` — existing GCP project to host the bucket
- `bucket_name` — globally unique GCS bucket name
- `region` — GCS bucket region

## Section 2 — GCP Baseline Module (`providers/gcp/modules/baseline/`)

A reusable Terraform module that configures a GCP project to a consistent baseline. Called once per project from `providers/gcp/projects/<name>/main.tf`.

### IAM & Identity
- Non-root admin user (personal Google account) bound to `roles/owner` at the project level
- Terraform operations authenticated via Workload Identity / ADC — no service account key files

### APIs
- Configurable list of GCP APIs to enable via `enabled_apis` variable
- Sensible defaults: `compute.googleapis.com`, `iam.googleapis.com`, `cloudbilling.googleapis.com`, `cloudresourcemanager.googleapis.com`
- Overridable per-project via tfvars

### Budget Alerts
- `google_billing_budget` resource tied to the project
- Billing account referenced via a `data` source (read-only — no-ops if already linked in console)
- Configurable variables:
  - `budget_amount` — monthly cap in USD
  - `budget_thresholds` — list of fractional alert thresholds (e.g. `[0.5, 0.9, 1.0]`)
- Email notification to admin user via Pub/Sub notification channel

### Labels
- Standard label set applied to the project resource
- Configurable via `labels` map in tfvars
- Suggested defaults: `env`, `owner`, `managed-by = terraform`

### Logging
- Audit log sink for admin activity and data access events
- Default destination: Cloud Logging (no extra bucket required); configurable to export to GCS if needed

### Defaults
- Default region and zone set via provider configuration in each project's `main.tf`

## Section 3 — Per-Project Instantiation (`providers/gcp/projects/my-project/`)

Thin wrapper that calls `module "baseline"` and configures the remote backend.

**`backend.tf`** — GCS backend pointing to bootstrap bucket with project-specific state prefix:
```
prefix = "gcp/my-project"
```

**`terraform.tfvars.example`** (committed):
```hcl
project_id        = "my-gcp-project-id"
billing_account   = "XXXXXX-XXXXXX-XXXXXX"
admin_user        = "user@gmail.com"
budget_amount     = 20
budget_thresholds = [0.5, 0.9, 1.0]
labels = {
  env        = "personal"
  owner      = "aditya"
  managed-by = "terraform"
}
enabled_apis = [
  "compute.googleapis.com",
  "iam.googleapis.com",
  "cloudbilling.googleapis.com",
  "cloudresourcemanager.googleapis.com"
]
```

**`terraform.tfvars`** — gitignored, contains real values.

## Section 4 — Pre-commit Hooks (`.pre-commit-config.yaml`)

Runs on every commit:
- `terraform fmt` — auto-formats all `.tf` files
- `terraform validate` — catches syntax and configuration errors
- `terraform-docs` — regenerates `README.md` in each module and project dir from variable/output definitions

Requires: `pre-commit`, `terraform-docs` installed locally.

---

## Section 5 — GitHub Actions (`.github/workflows/terraform.yml`)

Single reusable workflow parameterized by provider/project path.

**On pull request:**
1. `terraform fmt -check`
2. `terraform validate`
3. `terraform plan` — output posted as PR comment

**On merge to main:**
1. `terraform apply -auto-approve`

**Authentication:** GCP Workload Identity Federation for GitHub Actions — a Workload Identity Pool configured to trust `repo:your-org/core-infra:*` OIDC tokens. No service account keys stored as GitHub secrets.

**Scaling:** Matrix strategy fans out across multiple project dirs when additional GCP projects are added.

---

## Extensibility

- Adding a new GCP project: create `providers/gcp/projects/<name>/`, copy structure from `my-project/`, fill in tfvars.
- Adding AWS: create `providers/aws/modules/baseline/` and `providers/aws/projects/<name>/` following the same pattern.
- Module extraction: `providers/gcp/modules/baseline/` can be moved to a separate repo and referenced via `source = "git::https://..."` with minimal changes to project instantiation dirs.
